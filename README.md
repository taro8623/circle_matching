# Circle Matching

大学のバンドサークル向けに、**曲募集・パートマッチング・メンバー確定・定期ライブ申請・曲単位チャット**を一気通貫で扱うフルスタックアプリです。

単なる掲示板ではなく、**「誰がどの曲のどのパートに入るか」から「その曲をどのライブに出すか」までを業務フローとして整理している**のがこのアプリの特徴です。

現在の実装だけでも、**ユーザー登録 → サークル参加 → 担当パート設定 → 曲起票 → 応募 / お誘い → メンバー確定 → チャット → ライブ申請 → 通知確認**まで、一連の流れをローカル環境で動かせます。

## 公開デモ運用メモ

- 公開デモでは **新規ユーザー登録を停止中** です。
- テスト用ユーザーの共通パスワードは **`RootsDemo2026!`** です。
- デモ環境の **DB内容は定期的に消去 / 再投入** します。
- 個人情報や消したくないデータの登録は想定していません。
- お問い合わせ・機能要望は [Googleフォーム](https://docs.google.com/forms/d/e/1FAIpQLScPIYWK9lXV2DXWaxtnrlg07gb7_TuEP5Gq07Fj7OOPqUy9KA/viewform) から受け付けています。
- 新規登録を再開する場合は、バックエンドで `PUBLIC_SIGNUP_ENABLED=true`、フロントエンドで `VITE_PUBLIC_SIGNUP_ENABLED=true` を設定して再デプロイしてください。

## Render デプロイ

- `render.yaml` を用意してあるので、Render の **Blueprint** からそのまま作成できます。
- 作成される構成は **PostgreSQL + FastAPI API + Vite 静的フロント** です。
- API 起動時に `alembic upgrade head` と `python seed_roots_demo_users.py` を実行し、公開デモ用の `roots` サークルとテストユーザーを自動投入します。
- フロントは `https://circle-matching-web.onrender.com`、API は `https://circle-matching-api.onrender.com` を前提にしています。
- デモデータを戻したい場合は、バックエンド環境で `python reset_demo_data.py && python seed_roots_demo_users.py` を実行してください。

## このアプリで解決したい課題

バンドサークルでは、以下のような情報が分散しがちです。

- 曲ごとの募集状況
- 誰がどのパートをやりたいか
- 誰を誘うか、誰が承認したか
- 定期ライブへの出演予定
- メンバー確定後の連絡手段

このアプリではそれらを分断せず、**曲を中心にサークル活動を管理する**設計にしています。

## 現在できること

### 1. 認証 / マイページ

- メールアドレスとパスワードで新規登録（ローカルでは利用可能。公開デモでは停止中）
- ログイン時に JWT を発行
- フロント側でトークンを保持し、保護ルートを制御
- `/me` でログインユーザー情報を取得
- 自己紹介の登録 / 更新（プロフィール必須項目）
- 担当パートの登録 / 更新
- 自由入力のパート追加
- 参加中サークル一覧の表示
- ログアウト

### 2. サークル管理

- サークル作成
- 参加用パスワード付きでサークルに参加
- 既存サークルへの再参加
- サークル脱退（論理削除）
- サークル詳細の表示
- メンバー一覧の表示
- パート別のメンバー絞り込み
- メンバー一覧で他メンバーの自己紹介を閲覧
- owner / admin / member のロール管理

### 3. 曲募集 / 曲マッチング

- サークル内で新しい曲を起票
- 曲名、アーティスト、参考 URL、メモ、時期希望メモの登録
- パートごとに複数枠を設定
- 各枠を以下のモードで設定可能
  - 募集中
  - 自分で担当
  - サークル内メンバーを指名
  - 外部メンバーを登録
- Vo / Gt / Ba / Dr / Key / Cho / Other に加え、カスタムパートの追加
- 同一パート内での重複メンバー指定防止
- 曲詳細で募集人数と充足人数を確認
- 外部メンバーを含めた決定済みメンバー表示
- 曲情報の編集
- 曲ステータスの変更
  - recruiting
  - ready
  - archived
  - cancelled
- 予定月（planned month）の設定

### 4. 自分向けの曲発見 UX

- サークル詳細画面で、現在の自分に関係する曲を分類表示
  - 自分が起票した曲
  - 応募できる曲
  - 応募中の曲
  - お誘いを受けている曲
  - 参加決定済みの曲
- 自分の担当パートと募集パートの突合
- 募集中の曲を一覧で確認
- Tinder ライクなカード UI で「興味なし / 応募する」を操作

### 5. 応募 / お誘い / メンバー確定

- 募集中パートへの応募
- 起案者から特定メンバーへのお誘い
- 同一ユーザー・同一パートへのアクティブ重複応募防止
- 応募 / お誘いの状態管理
  - pending
  - accepted
  - declined
  - withdrawn
- 権限に応じた状態変更
  - 応募は起案者が承認 / 見送り
  - お誘いは誘われた本人が承認 / 辞退
  - 取り下げは適切な当事者のみ可能
- 曲ごとの応募 / お誘い一覧表示

### 6. 曲チャット

- 曲ごとのチャットルーム
- 応募やお誘いの発生時にチャット参加者を追加
- メッセージ投稿 / 一覧取得
- 参加者以外はチャット閲覧不可
- 曲単位で会話が残るため、ライブが変わっても文脈を維持できる

### 7. 定期ライブ管理

- サークル管理者による定期ライブ作成
- ライブ作成直後は申請受付 closed で開始し、準備できたら open に切り替え
- ライブ一覧表示
- ライブ受付の open / closed 切り替え
- ライブ状態の手動管理
  - scheduled
  - completed
  - cancelled
- 開催日を過ぎた未完了ライブへの終了処理警告
- 各ユーザーの月別参加意思表明
  - 誘ってほしい
  - 出演予定あり
  - 今月は出演予定なし
- 手動の参加意思とは別に、その月の出演確定 / 申請中 / 参加予定を自動集計表示
- ライブごとに申請中 / 承認済みの曲を表示
- ライブカードから曲詳細へ遷移

### 8. ライブ出演申請

- 起案者が曲を特定ライブへ申請
- 同一ライブへの重複申請防止
- 他サークルのライブへの申請防止
- 受付終了ライブへの申請防止
- 管理者による承認 / 却下
- 申請者による取り下げ
- 曲単位・ライブ単位の申請一覧取得

### 9. 通知

- お誘い受信通知
- 応募承認 / 見送り通知
- お誘い承認 / 辞退通知
- チャット新着通知
- ライブ申請の承認 / 却下通知
- 通知一覧表示
- 個別既読
- 一括既読
- 通知から関連画面へ遷移

## アプリケーションエンジニアとしての見どころ

このプロジェクトでは、UI を作るだけではなく、**業務フローをどうデータモデルへ落とすか**を重視しています。

### 1. ドメイン分解

以下を別レイヤーとして整理しています。

- サークル所属
- 曲募集
- 応募 / お誘い
- メンバー確定
- 定期ライブ申請
- 通知
- 曲単位チャット

特に、**応募とお誘いを `song_part_entries` に統合しつつ `kind` と `status` で制御している点**は、機能追加しやすさと履歴管理を意識した設計です。

### 2. 権限設計

- JWT ベースの認証
- サークル所属チェック
- owner / admin / member の権限制御
- 起案者だけができる操作と、参加者だけができる操作を API 側で明確に分離

フロントだけでなく、**バックエンドで権限を担保する実装**にしています。

### 3. 一貫したフルスタック実装

- React + TypeScript で画面実装
- FastAPI + SQLAlchemy で API / 永続化
- 画面遷移と API の役割分担を明確化
- 曲一覧、曲詳細、ライブ一覧、通知一覧など、複数画面にまたがる状態遷移を接続

### 4. プロダクト視点

このアプリは「技術を使った」だけではなく、**ユーザーの行動単位で画面と API を切っている**点が強みです。

たとえば、

- 自分が応募できる曲を先に見せる
- 誰を誘うかを曲詳細に集約する
- 曲が決まった後はそのままチャットへ遷移する
- さらにその曲をライブ申請につなげる

というように、**操作の連続性を意識して設計しています。**
